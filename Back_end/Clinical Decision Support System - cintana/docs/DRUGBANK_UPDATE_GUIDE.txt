DOKUMENTASI UPDATE DATASET DRUGBANK (CDSS)
Tanggal: 2025-12-30
Target rilis contoh: DrugBank Release Version 5.1.13 (contoh tabel rilis: 2025-01-02)

Ringkasan
- Project ini mengimpor DrugBank XML ke tabel `drugs_drug` (model `Drug`) menggunakan management command:
  `python manage.py load_drugbank <path_xml> [--reset] [--only DBxxxx] [--progress N]`
- Loader melakukan parsing bertahap (iterparse) dan `update_or_create()` per DrugBank ID.
- Loader juga menyimpan payload mentah setiap entri `<drug>` ke field `raw_xml` agar seluruh konten XML tetap tersimpan (lossless backup per-record).
- Tujuan SOP ini: memastikan update dataset (mis. rilis baru dari DrugBank) dapat dilakukan aman, terukur, dan bisa di-rollback jika terjadi masalah.

Lokasi & Komponen Penting
- Command importer: drugs/management/commands/load_drugbank.py
- Model target: drugs/models.py (model `Drug`, banyak JSONField)
- Contoh dataset lokal: drugbank_full.xml (root workspace)
- DB default di repo ini: db.sqlite3 (root workspace)

Catatan Lisensi (parafrase)
- Pastikan Anda memiliki hak akses dataset yang digunakan.
- Untuk “Academic Downloads” (umumnya gratis untuk akademik): biasanya mensyaratkan afiliasi institusi akademik, non-komersial, dan penggunaan untuk tujuan akademik.
- Open Data DrugBank umumnya berada di bawah CC BY-NC 4.0 (non-komersial) — cek ketentuan terbaru di DrugBank.
- Jangan mendistribusikan ulang file lengkap jika lisensi Anda tidak mengizinkan.
- Jika dipakai untuk publikasi/riset, sertakan sitasi DrugBank sesuai panduan sitasi resmi.

A. PRA-SYARAT
1) Python environment
- Gunakan venv project (contoh Windows):
  - Aktifkan: venv\Scripts\Activate.ps1
  - Pastikan dependensi: pip install -r requirements.txt
- Kompatibilitas DB: gunakan MySQL 8.0+ (direkomendasikan).

2) Pastikan migrasi sudah sesuai
- Jika tidak ada perubahan struktur model, cukup pastikan DB sudah migrate:
  - python manage.py migrate

3) Siapkan ruang disk dan waktu
- File DrugBank XML besar. Import full dapat memakan waktu menit–jam tergantung mesin.


B. CHECKLIST SEBELUM UPDATE
1) Backup database
- Jika memakai SQLite (db.sqlite3), lakukan backup file terlebih dahulu:
  - Tutup server/dev yang sedang memakai DB.
  - Copy file: db.sqlite3 -> backup/db.sqlite3.YYYYMMDD-HHMM

2) Simpan file XML lama
- Simpan versi sebelumnya (untuk audit/rollback):
  - drugbank_full.xml -> backups/drugbank_full.<versi>.<tanggal>.xml

3) Verifikasi file XML baru
- Pastikan file yang didownload adalah “Complete Database” (XML) sesuai lisensi.
- Cek sekilas root element dan namespace.
  Loader saat ini mendeteksi namespace otomatis dari root XML.

  Cara cek cepat (tanpa tool tambahan):
  - Buka beberapa baris awal XML dan cari:
    - atribut version / release
    - xmlns=...

4) Pastikan schema (opsional)
- Repo memiliki drugbank.xsd. Validasi XSD bersifat opsional (butuh tool/library khusus).


C. STRATEGI UPDATE (REKOMENDASI)
Pola aman yang disarankan:
1) Dry-run terbatas (pakai --only)
2) Full reload (pakai --reset) saat sudah yakin

Alasan:
- `update_or_create()` aman untuk update per record, tetapi tidak menghapus entri yang sudah tidak ada di rilis baru.
- Full reload memastikan DB hanya berisi data dari snapshot XML baru.


D. LANGKAH UPDATE (WINDOWS)
Catatan: gunakan path dengan tanda kutip jika ada spasi.

0) Cek cepat: namespace + metadata rilis dari XML (disarankan)
- Tujuan: memastikan `load_drugbank.py` membaca XML yang benar dan namespace tidak berubah.
- Jalankan dari root project:
  python -c "import xml.etree.ElementTree as ET; p='drugbank_full.xml'; it=ET.iterparse(p, events=('start',)); ev, root=next(it); tag=root.tag; ns=tag.split('}')[0].strip('{') if tag.startswith('{') else ''; print('root_tag:', tag); print('namespace:', ns or '(none)'); print('root_attrib_keys:', sorted(root.attrib.keys())); print('root_attrib:', root.attrib)"

  Interpretasi:
  - Jika `root_tag` mirip `{http://www.drugbank.ca}drugbank`, maka namespace XML adalah `http://www.drugbank.ca`.
  - Jika output namespace berbeda, loader tetap mencoba menyesuaikan otomatis.
  - `root_attrib` bisa berisi metadata seperti versi/export date (tergantung rilis). Simpan nilainya di logbook.

1) (Opsional) Buat folder backup
- backup/
- backups/

2) Taruh file XML baru
- Letakkan file XML baru di root workspace (contoh):
  - drugbank_full.xml
  atau simpan dengan nama versi:
  - drugbank_full_5.1.13.xml

3) Dry-run terbatas (sangat disarankan)
- Pilih 1–3 DrugBank ID yang Anda kenal ada di dataset (contoh: DB00421).
- Jalankan:
  python manage.py load_drugbank "drugbank_full.xml" --only DB00421 --progress 1

  Output yang diharapkan:
  - seen bertambah
  - saved minimal 1
  - tidak ada error fatal

4) Full reload (mode paling aman)
- Jalankan (contoh progress log tiap 500):
  python manage.py load_drugbank "drugbank_full.xml" --reset --progress 500

  Penjelasan parameter:
  - --reset   : hapus semua data Drug lalu import ulang
  - --progress: interval log progress (mis. 200 atau 500)

5) (Opsional) Incremental update (tanpa reset)
- Jika Anda sengaja ingin mempertahankan data lama dan hanya update yang matching ID:
  python manage.py load_drugbank "drugbank_full.xml" --progress 500


E. VERIFIKASI PASCA-IMPORT
1) Cek jumlah row
- Setelah command selesai, command sudah mencetak jumlah row.
- Anda juga bisa cek manual:
  python manage.py shell
  >>> from drugs.models import Drug
  >>> Drug.objects.count()

2) Cek sampel record penting
- Pilih 2–5 drug populer dan pastikan field-field kunci terisi:
  - name, description
  - smiles (untuk small molecule)
  - targets / enzymes / transporters / carriers (JSON)
  - interactions, food_interactions
  - genomics
  - general_references, products

3) Smoke test halaman
- Jalankan server lalu cek:
  - / (search)
  - /drug/<DBID>/
  - /interaction-checker/
  - /simulation/


F. TROUBLESHOOTING (KESALAHAN YANG SERING TERJADI)
1) “saved=0” atau data banyak kosong
- Penyebab umum: file XML tidak sesuai format DrugBank expected atau rusak.
- Gejala: node penting tidak terbaca walau namespace sudah auto-detect.
- Solusi:
  - Cek kembali file XML sumber.
  - Jalankan dry-run `--only` untuk validasi satu ID.

2) Proses sangat lambat
- File besar + banyak JSON extraction.
- Solusi:
  - Naikkan --progress untuk mengurangi log spam.
  - Jalankan di mesin yang lebih cepat.
  - Pastikan tidak ada proses lain yang berat.

3) Memory tinggi
- Loader sudah memakai iterparse dan elem.clear() untuk hemat memori.
- Jika masih berat, pastikan tidak ada modifikasi yang menghapus elem.clear().

4) Data “tertimpa” oleh entri stub
- Loader sudah mencoba skip entri tanpa atribut `type` (stub) agar tidak overwrite data yang lebih kaya.


G. ROLLBACK PLAN (JIKA TERJADI MASALAH)
1) Stop aplikasi/server
2) Restore DB backup
- Replace db.sqlite3 dengan file backup yang dibuat pada tahap pra-update.
3) Kembalikan file XML lama (jika diperlukan)
4) Re-run import menggunakan file yang benar


H. CATATAN TEKNIS LOADER (UNTUK MAINTAINER)
- Importer menggunakan xml.etree.ElementTree.iterparse(..., events=('end',))
- Memproses setiap <drug> lalu melakukan:
  Drug.objects.update_or_create(drugbank_id=db_id, defaults={...})
- Struktur JSON yang dihasilkan (ringkas):
  - categories: [{"category": str, "mesh_id": str|null}, ...]
  - interactions: [{"drugbank_id": str, "name": str, "description": str}, ...]
  - targets/enzymes/transporters/carriers:
    [{"name": str, "organism": str, "actions": [str], "gene_name": str, "uniprot_id": str|null, "function": str|null, ...}, ...]
  - genomics:
    [{"gene": str, "variant": str|null, "type": "Effect"|"Adverse", "description": str}, ...]
  - general_references: {articles/textbooks/links/attachments: [...]}
  - products/international_brands/external_links/external_identifiers/sequences: list of dict
- Untuk jaminan cakupan penuh, field `raw_xml` menyimpan seluruh elemen `<drug>` asli per record.
